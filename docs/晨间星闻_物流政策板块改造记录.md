# 晨间星闻物流与政策板块改造记录

## 一、改造背景

本次改造基于两个连续需求：

1. 将原有“运小星日报”升级为更贴合跨境 AI 科技实时热点的晨报型产品，名称调整为“晨间星闻”。
2. 在不改动 AI 板块数据逻辑的前提下，重新梳理物流板块和政策板块的操作方式。

后续补充要求：

- 不能改变原有视觉背景。
- 必须恢复并保留原来的“运小星 + 百运网蓝色科技风”背景。
- 页面可以叫“晨间星闻”，但视觉底色、品牌感和蓝色运小星 banner 不应被替换成新的风格。

因此，最终处理原则为：

> 视觉沿用原运小星蓝色背景；AI 板块不改；物流板块和政策板块按跨境物流业务影响重新分拣。

## 二、用户提供的新政策情报 Prompt 要点

用户提供的 prompt 定位为：

> 跨境国际物流政策情报监控 Agent

目标用户包括：

- 销售
- 客服
- 操作
- 产品
- 管理层

核心原则如下：

1. 官方源优先，媒体源辅助。
2. 不允许臆造信息，所有结论必须能追溯到原文链接。
3. 政策类信息必须提取发布时间、生效时间、涉及国家、涉及品类、涉及 HS 编码、政策变化、物流影响。
4. 行业新闻类信息必须判断是否影响运价、时效、清关、禁限运、航线、港口、派送、客户报价。
5. 对跨境物流业务无明显影响的信息可以忽略。
6. 来源不可靠的信息只能标记为“待确认”，不能直接作为正式结论。
7. 输出必须让业务人员看得懂，不能只复制政策原文。

## 三、三大板块定义

### 1. 科技动态

AI 板块不改动。

数据来源仍使用现有 AI HOT API：

- AI 模型
- AI 产品
- AI 工具
- AI 论文
- AI 行业观点

前端仅将该板块展示名称统一为“科技动态”，但不修改 AI 抓取、AI 分类和 AI 处理逻辑。

### 2. 行业动态

行业动态只处理跨境物流和业务执行层面的内容。

应进入行业动态的内容包括：

- 运价变化
- 空运、海运、快递、专线动态
- 航线、港口、船司、仓位、附加费
- 末端派送、海外仓、FBA、平台履约
- 清关实操、申报资料准备、查验应对
- 跨境平台规则对发货和利润的影响
- 百运百科中的实操型物流知识

不应进入行业动态的内容：

- 纯关税政策
- 官方海关规则变更
- 制裁、出口管制、实体清单
- 重大地缘政治风险

这类内容应进入政策动态。

### 3. 政策动态

政策动态只处理会影响跨境物流业务的政策、监管和地缘风险。

应进入政策动态的内容包括：

- 关税政策
- 美国对华贸易政策
- Section 301、Section 232
- 反倾销、反补贴、贸易救济
- 海关清关规则、ACE、CBP、申报要求
- 出口管制、制裁、实体清单、限制名单
- 原产地、HS 编码、低值免税、税则调整
- 红海、海峡、封锁、战争、空域关闭、港口封锁等地缘风险

政策动态必须输出更完整的业务字段：

- 发布时间
- 生效时间
- 涉及国家或地区
- 涉及业务线
- 涉及 HS 编码
- 涉及品类
- 政策或事件摘要
- 关键变化
- 对跨境物流的影响
- 销售/客服提醒话术
- 操作/清关提醒
- 风险等级
- 是否需要更新知识库
- 是否需要人工复核

## 五、前端分拣规则

前端页面 `/daily` 统一展示为“晨间星闻”，但保留原有运小星蓝色 banner。

页面展示四个筛选入口：

- 全部
- 科技动态
- 行业动态
- 政策动态

前端分类逻辑如下：

1. AI HOT 返回的数据全部进入“科技动态”。
2. policy-official 任务下的数据默认进入“政策动态”。
3. global-news 任务下的数据根据关键词判断：
   - 涉及关税、制裁、出口管制、实体清单、地缘风险、红海、战争、封锁等，进入“政策动态”。
   - 仅涉及油价、能源成本或市场背景，但没有明确政策/地缘风险的，可进入“行业动态”或作为低风险观察。
4. logistics-daily、crossborder-platform、shipping-port、by56-wiki 默认进入“行业动态”。
5. 如果行业源中出现明显政策关键词，例如关税、反倾销、CBP、USTR、出口管制、制裁、地缘冲突，也会被前端归入“政策动态”。

## 八、后续建议

### 1. 政策源需要进一步稳定化

目前第一版优先加入可用网页源。后续如果要更稳定，应单独处理 API 型或数据库型来源，例如：

- Federal Register API
- Trade.gov Consolidated Screening List API
- GOV.UK Trade Tariff API
- WTO Tariff & Trade Data
- WITS
- EU TARIC
- Access2Markets

这些不建议直接用普通网页抓取，应单独做 API 适配。

### 2. 政策动态建议加人工复核状态

政策动态涉及关税、生效时间、HS 编码和客户报价，不适合完全自动发布。建议后续增加：

- 待复核
- 已确认
- 已同步销售/客服
- 已更新知识库

### 3. 行业动态和政策动态建议分开缓存

当前仍复用 `/api/briefing/logistics` 和 `logistics_cache.json`。后续更清晰的结构应拆成：

- `/api/briefing/industry`
- `/api/briefing/policy`
- `industry_cache.json`
- `policy_cache.json`

第一版为减少改动，暂时没有拆接口。

## 十一、刚才用户重点要求的处理结论

1. “晨间星闻”是当前栏目/产品名称。
2. 原来的运小星、百运网、蓝色科技风背景必须保留；当前 hero 以用户指定的“晨间星闻横幅”为准。
3. AI 板块只改展示归属为“科技动态”，数据逻辑不动。
4. 物流板块改为“行业动态”，负责物流、平台、运价、时效、航线、港口、清关实操。
5. 政策板块改为“政策动态”，负责关税、海关规则、制裁、出口管制、实体清单、地缘政治风险。
6. 国外政策和英文物流来源可以在推送前转成中文，但必须保留原文链接、机构名、法规编号、HS 编码、日期、金额等关键事实。
7. 后续所有新增修改都应继续写入本文档，避免口径丢失。
